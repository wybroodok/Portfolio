# CosApp — шаблон Telegram-бота для записи на услуги

Готовый шаблон Telegram-бота с мини-приложением (Mini App) для салонов красоты,
барбершопов, мастеров маникюра/массажа и любого другого бизнеса, работающего по
записи. Клиент записывается, видит свои записи, акции и оставляет отзывы —
всё внутри Telegram, без отдельного сайта. Администратор управляет услугами,
слотами времени и акциями там же, в мини-аппе, без панели где-то на сервере.

Этот README — единственная документация, которая нужна, чтобы взять шаблон и
запустить свой собственный бот под свой бизнес. Если что-то не описано здесь —
это обычно означает "просто прочитай код в указанном файле", и в самом коде
щедро оставлены комментарии на этот случай.

## Что это и как устроено

Технически это два взаимосвязанных проекта, запускаемых одним процессом:

| Часть | Технология | Папка |
|---|---|---|
| Telegram-бот | Python 3.14, [aiogram](https://docs.aiogram.dev/) 3.29 | корень репозитория |
| API для мини-аппа | FastAPI поверх той же базы данных | `api/app.py` |
| Мини-апп (фронтенд) | React 19 + TypeScript + Vite + Tailwind CSS v4 | `webapp/` |
| База данных | SQLite (файл `cosapp.db`, создаётся автоматически) | корень репозитория |

Бот и API-сервер живут в одном asyncio-процессе (`main.py`) и работают с одной
и той же базой данных напрямую — никакого HTTP-моста между ними нет.
Мини-апп — это отдельное веб-приложение, которое открывается внутри Telegram
(через кнопку меню бота) и ходит в FastAPI по HTTP.

**Вся функциональность — и клиентская, и админская — реализована в мини-аппе.**
У бота в чате остаётся только `/start` с приветствием: вся работа (запись,
"Мои записи", отзывы, акции, админ-панель) — экраны мини-аппа. Так исторически
было по-другому (были ещё и текстовые команды/кнопки в чате бота), но сейчас
это всё удалено в пользу единой точки входа — см. `handlers/start.py`.

## Быстрый старт

Понадобится: Python 3.11+ (в разработке использовался 3.14), Node.js 20+,
токен Telegram-бота от [@BotFather](https://t.me/BotFather).

### 1. Бэкенд (бот + API)

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env
# открой .env и впиши BOT_TOKEN и ADMIN_IDS (см. ниже "Настройка под свой бизнес")
```

Запуск:

```bash
python main.py
```

При первом запуске автоматически создастся файл `cosapp.db` со всеми таблицами
(пустыми) — заполнять его руками не нужно, он появляется сам. Если всё
прошло успешно, в терминале появится `Run polling for bot @твой_бот`.

### 2. Фронтенд (мини-апп) — локальная разработка

```bash
cd webapp
npm install
npm run dev
```

Откроется на `http://localhost:5173`. Vite сам проксирует все запросы `/api/*`
на `http://localhost:8000` (см. `webapp/vite.config.ts`), так что в браузере
(без Telegram) можно потыкать интерфейс уже сейчас — запись будет работать в
анонимном режиме (см. ["Как всё это работает изнутри"](#авторизация)).

Но чтобы увидеть его **по-настоящему**, как Telegram Mini App (с именем
пользователя, аватаркой, кнопками Telegram) — приложению нужен публичный
HTTPS-адрес и настройка в @BotFather. Это — раздел
["Деплой"](#деплой-чтобы-бот-работал-не-только-на-твоём-компьютере) ниже.

### 3. Первые шаги после запуска

1. Напиши своему боту `/start` в Telegram — если ID совпадает с одним из
   `ADMIN_IDS` в `.env`, у тебя автоматически будут права администратора.
2. Открой мини-апп → **Настройки** → **Админ-панель**.
3. Зайди в **Услуги** и добавь хотя бы одну услугу (название, длительность,
   цена) — без этого раздел "Записаться" будет пустым.
4. Зайди в **Слоты** и добавь несколько доступных дат/времени.
5. Всё, можно записываться: **Главная → Записаться**.

## Настройка под свой бизнес

Всё, что отличает "твой" салон от чужого — название, описание, контакты,
баннер, ID администратора — вынесено в **два места**. Больше никакой код
трогать не нужно.

### `webapp/src/siteConfig.ts` — тексты и внешний вид

```ts
export const BUSINESS_NAME = 'Студия красоты'
export const BUSINESS_TAGLINE = 'ваш уход в лучших руках'
export const ABOUT_DESCRIPTION = 'Профессиональный уход за кожей. ...'
export const SPECIALISTS_TITLE = 'Наши специалисты'
export const SPECIALISTS_TEXT = 'Опытные косметологи ...'   // пустая строка — блок не показывается
export const CONTACTS = {
  address: 'ул. Красоты, 12, Москва',   // пустая строка — строка не показывается
  phone: '+7 (999) 123-45-67',
  hours: 'Пн–Сб: 9:00–20:00',
  instagram: '@beauty_studio',
}
export const HOME_BANNER = {
  imageUrl: '',   // ссылка на картинку (или файл из webapp/public/) — пусто = баннера нет
  linkUrl: '',    // куда ведёт тап по баннеру (необязательно)
}
export const MASTER_CONTACT_URL = ''  // https://t.me/username — пусто = кнопки нет
```

Где что появляется:

| Константа | Где видно в приложении |
|---|---|
| `BUSINESS_NAME`, `BUSINESS_TAGLINE` | Подзаголовок на главной странице ("Привет Гость! — *Студия красоты — ваш уход в лучших руках*"), заголовок в "О нас" |
| `ABOUT_DESCRIPTION` | Абзац-описание в карточке "О нас" (и на отдельной странице, и в "Настройках" — контент общий) |
| `CONTACTS.*` | Список адрес/телефон/часы/Instagram в "О нас" |
| `SPECIALISTS_TITLE` / `_TEXT` | Нижний блок в "О нас" |
| `HOME_BANNER` | Картинка-баннер под 4 карточками на главной странице |
| `MASTER_CONTACT_URL` | Кнопка "Связаться с мастером" в "Настройках" (открывает личку в Telegram через `openTelegramChat`, см. `webapp/src/telegram.ts`) |

**Баннер:** картинку можно положить в `webapp/public/` (например
`my-banner.jpg`) и указать `imageUrl: '/my-banner.jpg'`, либо указать любой
внешний URL картинки. Рекомендуемое соотношение сторон — 2:1 (например
800×400) — баннер всегда показывается в этой пропорции с обрезкой
(`object-cover`), независимо от реального размера файла.

После правки `siteConfig.ts` фронтенд нужно пересобрать (см. "Деплой") —
на dev-сервере (`npm run dev`) изменения подхватываются сами.

### `.env` — секреты и техническая конфигурация

Скопируй `.env.example` в `.env` и заполни:

| Переменная | Что это | Обязательна? |
|---|---|---|
| `BOT_TOKEN` | Токен бота от @BotFather (`/newbot` → скопировать токен) | да |
| `ADMIN_IDS` | Твой числовой Telegram ID (и/или ID других администраторов через запятую, без пробелов) — узнать свой ID можно у бота [@userinfobot](https://t.me/userinfobot) | да |
| `TIMEZONE` | Часовой пояс (влияет на "сегодня" в блокировке дня, время напоминаний и т.п.) — любой ключ IANA/zoneinfo, например `Europe/Moscow`, `Asia/Yekaterinburg`, `Europe/Madrid` | нет, по умолчанию `Europe/Moscow` |
| `WEBAPP_URL` | Публичный HTTPS-адрes, где открывается мини-апп (см. "Деплой") — используется для кнопки "Открыть приложение" под уведомлениями бота | нет, но без него уведомления идут без кнопки |
| `API_PORT` | Порт, на котором слушает FastAPI локально | нет, по умолчанию `8000` |

**Услуги, цены, время работы записи (слоты) — это не конфиг, а данные.** Они
вводятся не в файлах, а через саму админ-панель в приложении (Услуги/Слоты) —
это то, что администратор салона обычно меняет сам, без разработчика, и часто.

## Как всё это работает изнутри

### Авторизация

Мини-апп получает данные пользователя от Telegram через `window.Telegram.WebApp.initData` —
подписанную HMAC-строку (секрет — тот же `BOT_TOKEN`). Сервер (`api/app.py`,
функция `_validate_init_data`) проверяет подпись на каждый запрос: если
подпись верна — сервер доверяет присланному `user.id`; если нет — считает
запрос неавторизованным. Никаких паролей и сессий: доверие полностью
завязано на криптографическую проверку initData.

Если приложение открыто **вне** Telegram (например, в обычном браузере при
разработке) — initData нет, и часть функций работает в "анонимном" режиме:
ID записи, которые сделал такой посетитель, сохраняются в `localStorage`
браузера (см. `webapp/src/pages/Booking.tsx` и `MyBookings.tsx`) — так можно
пользоваться разделом "Мои записи" без Telegram вообще, хоть это и не
основной сценарий использования.

Права администратора проверяются на сервере (`_is_admin`) — сверкой ID из
initData со списком `ADMIN_IDS` из `.env`. Пункт "Админ-панель" в
"Настройках" виден всем, но при заходе сама панель (`AdminGuard.tsx`) либо
пускает, либо показывает "Извините, вы не администратор" — то есть доступ
не скрыт, а по-настоящему проверен и на фронтенде (для UX), и на бэкенде
(для безопасности; проверка на фронте сама по себе ничего не защищает).

### Жизненный цикл записи (bookings)

1. **Слот** (`time_slots`) создаётся администратором на конкретные дату и
   время, `is_available = true`.
2. Клиент выбирает услугу → дату (интерактивный календарь) → время из
   свободных слотов на эту дату → вводит имя и телефон → подтверждает.
3. `POST /api/bookings` **атомарно** занимает слот (`UPDATE ... WHERE
   is_available = TRUE`, проверка `rowcount`) — так что даже если два
   человека одновременно жмут "Подтвердить" на один и тот же слот, второй
   гарантированно получит ошибку 409, а не удвоение записи. Плюс проверка,
   что у этого же пользователя ещё нет своей записи на то же время.
4. Запись создаётся со статусом `pending` ("Ожидает"). Администратор в
   Админ-панели → Все записи может **подтвердить** (`confirmed`) или
   **отменить** (`cancelled`, слот освобождается). Клиент тоже может
   отменить свою запись сам в любой момент через "Мои записи".
5. При подтверждении планируется автоматический **запрос отзыва**
   (`review_requests`) — через `длительность_услуги + 2 часа` после начала
   визита.
6. Каждую минуту фоновый планировщик (`scheduler.py`) проверяет: не пора ли
   прислать (а) запрос отзыва, (б) напоминание за ~24 часа до визита, (в)
   напоминание, что акция скоро закончится. Клиент отвечает прямо в
   мини-аппе (подтвердить визит, оставить отзыв, отменить запись).
7. **"Я сегодня занят" (форс-мажор)** в Админ-панели одним нажатием отменяет
   все ещё не начавшиеся записи на сегодня и блокирует остаток дня для новых
   записей — с уведомлением каждого затронутого клиента.

### Акции (promotions)

Администратор создаёт акцию (текст и/или фото, дата окончания) в
Админ-панели → Акции — она сразу рассылается **всем**, кто хоть раз открывал
бота (таблица `bot_users`), и появляется в разделе "Акции" у клиентов до
истечения `end_date`. За 2 дня до окончания планировщик разошлёт разовое
напоминание. Редактирование уже опубликованной акции **не** рассылает её
заново — это осознанно, чтобы правка опечатки не выглядела спамом.

### Уведомления администратору

Админы получают личные сообщения от бота о: новой записи, подтверждении
визита клиентом, новом отзыве. Отправка не блокирует ответ пользователю —
если бот не может написать одному из админов (например, тот не запускал
бота), это тихо игнорируется и не мешает остальному.

## База данных: что где хранится

Файл `cosapp.db` — обычный файл SQLite в корне репозитория, в `.gitignore` (не
попадает в git). Открыть и посмотреть его можно, например,
[DB Browser for SQLite](https://sqlitebrowser.org/) или консольным клиентом:

```bash
sqlite3 cosapp.db
sqlite> .tables
sqlite> SELECT * FROM bookings;
```

Таблицы создаются автоматически при старте (`database/db.py: init_db()`) —
ничего руками накатывать не нужно. Если структура таблиц меняется в будущих
версиях кода, `init_db()` умеет накатывать простые миграции (`ALTER TABLE ...`)
поверх уже существующей базы, не теряя данные.

| Таблица | Что хранит | Кто пишет |
|---|---|---|
| `services` | Каталог услуг: название, описание, длительность, цена | админ (Услуги) |
| `time_slots` | Доступное время записи: дата/время, свободен ли (`is_available`) | админ создаёт; система занимает/освобождает |
| `bookings` | Записи клиентов: кто, на что, когда, статус (`pending`/`confirmed`/`cancelled`) | клиент создаёт; клиент/админ меняют статус |
| `review_requests` | Запланированные (ещё не отправленные) запросы отзыва | система (после подтверждения записи) |
| `reviews` | Оценки (1–5) и комментарии клиентов | клиент |
| `bot_users` | Все, кто хоть раз открывал бота — список для рассылок | система, при каждом заходе в мини-апп |
| `promotions` | Акции: текст, фото (Telegram `file_id`), дата окончания | админ (Акции) |
| `booking_logs` | Аудит-лог: кто, когда и что сделал с каждой записью | система, на каждое действие |

Все "содержательные" данные (кто когда записался, что купил, что написал в
отзыве) живут только в этих таблицах — свежий клон репозитория без `.env` и
без `cosapp.db` полностью пуст: ни одной услуги, ни одной записи, ни одного
пользователя. Всё это заполняется через саму админ-панель после первого
запуска (см. "Быстрый старт").

## Админ-панель — что есть и как пользоваться

Открывается: мини-апп → **Настройки** → **Админ-панель** (виден любому, но
пускает внутрь только тех, чей Telegram ID есть в `ADMIN_IDS`).

| Раздел | Что делает |
|---|---|
| **Все записи** | Список всех записей клиентов (любого статуса). Подтвердить / отменить / безвозвратно удалить. |
| **Слоты** | Список будущих слотов времени + форма добавления нового. Редактирование времени и удаление — только для ещё свободных слотов (если на слот уже есть запись, сначала нужно её отменить — иначе клиент бы незаметно "потерял" своё время). Кнопка **Выбрать** — отметить несколько слотов галочками и удалить все разом. |
| **Услуги** | Список услуг + форма добавления/редактирования. Удалить услугу, на которую есть хоть одна запись (даже старая/отменённая), нельзя — иначе сломалась бы история этой записи. Тоже есть массовый выбор и удаление. |
| **Отзывы** | Список всех отзывов со средним рейтингом + экспорт в CSV (открывается в Excel, кодировка с BOM). |
| **Акции** | Список всех акций (включая уже истёкшие) + форма создания новой (с фото, до 1024 символов текста — ограничение Telegram на подпись к фото) и рассылка всем пользователям бота. Редактирование/удаление без повторной рассылки. Массовый выбор и удаление. |
| **Я сегодня занят** | Форс-мажор: отменяет все ещё не начавшиеся записи на сегодня, блокирует остаток дня, уведомляет каждого затронутого клиента. |

## Деплой (чтобы бот работал не только на твоём компьютере)

Есть два рабочих варианта. Оба варианта всё равно требуют, чтобы Python-часть
(`python main.py`) была где-то постоянно запущена — это и бот, и API, и
планировщик одновременно, в одном процессе.

### Вариант А — всё через один сервер (проще всего)

FastAPI умеет сам раздавать уже собранный фронтенд (см. конец `api/app.py`,
раздел "Раздача статики webapp") — то есть отдельный хостинг для фронтенда не
нужен, достаточно одного публичного HTTPS-адреса на сам `main.py`.

```bash
cd webapp
npm run build          # соберёт webapp/dist/
cd ..
python main.py          # теперь отдаёт и API, и собранный фронтенд на одном порту
```

Дальше нужен публичный HTTPS-адрес, указывающий на этот порт (Telegram Mini
App **обязательно** требует HTTPS, plain HTTP не подойдёт):
- для теста — туннель, например [ngrok](https://ngrok.com/) (`ngrok http 8000`)
  или Cloudflare Tunnel;
- для постоянной работы — обычный VPS с доменом и HTTPS-сертификатом
  (например через Caddy/nginx + Let's Encrypt) проксирующий на порт `API_PORT`.

Полученный HTTPS-адрес пропиши в `.env` как `WEBAPP_URL` и укажи в
@BotFather (см. ниже).

### Вариант Б — фронтенд отдельно на Netlify, бэкенд отдельно

Так фронтенд получает свой стабильный бесплатный адрес на Netlify, а туннель
(ngrok и т.п.) нужен только для бэкенда:

```bash
cd webapp
./build-netlify.sh https://твой-туннель-или-домен-бэкенда.example.com
```

Скрипт вшивает адрес бэкенда прямо в собранные JS-файлы (переменная
`VITE_API_URL` на этапе сборки) — после сборки папку `webapp/dist/`
перетаскиваешь на [netlify.com](https://app.netlify.com/) (drag & drop, без
аккаунта разработчика/CLI). **Важно**: если адрес бэкенда меняется
(бесплатный ngrok выдаёт новый при каждом перезапуске) — нужно пересобрать
`dist/` этим же скриптом и перезалить на Netlify.

### Настройка в @BotFather

1. `/mybots` → выбрать бота → **Bot Settings** → **Menu Button** → **Configure
   menu button** → указать тот же HTTPS-адрес, что и `WEBAPP_URL` в `.env`.
   Это и есть кнопка, которой пользователи открывают мини-апп из чата с
   ботом (Telegram сам показывает её рядом с полем ввода).
2. Опционально: `/setdescription`, `/setabouttext`, `/setuserpic` — обычные
   настройки профиля бота, не связаны с этим шаблоном напрямую.

## Известные ограничения и куда копать дальше

Это не баги, а осознанные упрощения шаблона — если нужно другое поведение,
вот где смотреть:

- **Телефон валидируется только в российском формате** (`+7XXXXXXXXXX`) — и
  на клиенте (`webapp/src/pages/Booking.tsx`, функция `isValidPhone`), и на
  сервере (`api/app.py`, класс `BookingCreate`, поле `phone`). Для другой
  страны — поправить регулярку/логику в обоих местах (сервер — источник
  истины, проверка на клиенте только для UX).
- **Часовой пояс один на весь бизнес** (`TIMEZONE` в `.env`) — если нужно
  учитывать часовой пояс каждого клиента индивидуально, это существенно
  больший объём изменений (сейчас все даты/время в БД — "наивные", в одном
  общем поясе).
- **Один язык интерфейса** — весь текст на русском, захардкожен в компонентах
  (кроме бизнес-данных, которые в `siteConfig.ts`). Интернационализация не
  реализована.
- **Один тип рассылки** — акции уходят всем, кто открывал бота, без
  сегментации аудитории.
- Если нужно больше технических деталей о конкретном эндпоинте или функции —
  они хорошо прокомментированы прямо в коде (`api/app.py`,
  `database/requests.py`) объяснениями "почему так", а не просто "что
  делает" — начните оттуда.
